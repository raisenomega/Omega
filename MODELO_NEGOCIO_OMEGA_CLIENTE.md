## 1. FILOSOFÍA DEL MODELO

```
El cliente PYME entra pagando directo a OMEGA.
Disfruta el potencial completo de la plataforma — limitado por volumen, no por
funcionalidad bloqueada (excepto en Básico donde Analytics, Crisis Room,
Calendario y Media se activan al subir a PRO).

El objetivo principal detrás de cada interacción:
  (1) Capturar comportamiento real → base de datos de decisiones OMEGA
  (2) Demostrar valor → el cliente renueva y sube de plan
  (3) Cuando escala → se convierte en Reseller naturalmente

La PYME no es el destino final — es el punto de entrada al ecosistema OMEGA.
ARIA es el sistema nervioso invisible que hace posible los 3 objetivos anteriores.
```

## 8. LÍMITES DE CUENTAS Y ANTI-FRAUDE

### Límite de cuentas por plan

```
BÁSICO    → 1 cuenta por red (IG · FB · TikTok · LinkedIn)
PRO       → 2 cuentas por red
Add-on    → +1 cuenta extra por $15/mes (temporal · ese mes)

REGLA CRÍTICA: Las cuentas adicionales son SIEMPRE de la misma marca/negocio.
Si el cliente quiere gestionar cuentas de TERCEROS → es Reseller, no cliente PYME.
```

### Trigger de conversión a Reseller

```
Al intentar agregar cuenta social:
  → Sistema pregunta: "¿Esta cuenta es de tu negocio?"
  → SÍ → puede agregar como cuenta extra ($15/mes add-on)
  → NO → pantalla de upgrade a Reseller
         "Para gestionar cuentas de clientes necesitas el plan Reseller"
```

### Sistema Anti-Fraude — 6 capas

```
CAPA 1 · Email
  1 email = 1 cuenta PYME · bloqueo inmediato si intenta duplicar
  Período de adopción = 1 solo intento por email · siempre

CAPA 2 · Tarjeta (Stripe Fingerprint)
  Misma tarjeta en 2+ cuentas → alerta automática + bloqueo
  No se puede crear segunda cuenta con misma tarjeta

CAPA 3 · IP + Device Fingerprint
  Al registrar: IP · browser fingerprint · OS · timezone · user agent
  Misma IP con 3+ cuentas en 30 días → flag automático
  Mismo device fingerprint en 2+ cuentas → alerta SENTINEL

CAPA 4 · Cuentas sociales (más poderosa)
  Una cuenta IG/FB/TikTok/LinkedIn NO puede estar en 2 usuarios
  Unique constraint en DB: social_account_id → 1 solo cliente
  Intento de duplicar → bloqueo inmediato + alerta SENTINEL

CAPA 5 · Comportamiento (ARIA detecta)
  Similitud de prompts > 85% entre cuentas distintas → caso SENTINEL
  Mismo nicho + ciudad + keywords en cuentas distintas → flag
  Brand voice idéntico entre cuentas → evidencia de fraude

CAPA 6 · Límite adopción por IP
  Máximo 2 trials gratuitos por IP en 90 días
  El tercero requiere tarjeta aunque sea "gratis"
```

### Respuesta automática por severidad

```
1 señal débil (IP compartida)
  → Registrar · puede ser legítimo (familia · oficina · café)

2 señales medias
  → Banner "Verificación pendiente" · ARIA pide confirmar identidad

3+ señales o 1 señal fuerte (misma tarjeta · misma cuenta IG)
  → Cuenta suspendida preventivamente
  → Email: "Detectamos actividad inusual · contacta soporte"
  → Alerta a Ibrain en SENTINEL

Fraude confirmado
  → Cuenta cancelada sin reembolso
  → Blacklist: email + tarjeta + device fingerprint
  → Stripe disputa marcada
```

### Tabla SQL requerida

```sql
-- supabase/migrations/00004_anti_fraud_signals.sql
CREATE TABLE anti_fraud_signals (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         uuid REFERENCES auth.users(id),
  signal_type     text CHECK (signal_type IN (
                    'ip_match', 'card_match', 'device_match',
                    'social_account_match', 'aria_behavior_match')),
  matched_user_id uuid REFERENCES auth.users(id),
  confidence      int CHECK (confidence BETWEEN 0 AND 100),
  evidence        jsonb DEFAULT '{}',
  status          text DEFAULT 'pending'
                    CHECK (status IN ('pending','reviewed','confirmed_fraud','false_positive')),
  reviewed_by     text,
  reviewed_at     timestamptz,
  created_at      timestamptz DEFAULT now()
);

-- Unique constraint: 1 cuenta social = 1 usuario
ALTER TABLE social_accounts
  ADD CONSTRAINT uq_social_account_per_user
  UNIQUE (platform, platform_account_id);

-- RLS
ALTER TABLE anti_fraud_signals ENABLE ROW LEVEL SECURITY;
CREATE POLICY "solo_service_role" ON anti_fraud_signals
  USING (auth.role() = 'service_role');
```
